인터럽트

컴퓨터 게임을 하는 도중, 키보드나 마우스가 작동하지 않는다면 어떨까? 아니면 컴퓨터가 에러가 나서 작업관리자를 켜야하는데 작업관리자가 켜지지 않는다면 어떨까? 컴퓨터를 사용하는 사용자 입장에서 입출력 기능은 다른 무엇보다도 사용자경험(UX)에 큰 영향을 준다. 그렇다면 어떻게 CPU는 작업을 처리하면서 동시에 입출력 처리도 할 수 있을까? 지금부터 인터럽트의 개념과 처리과정을 알아보도록 하자.

인터럽트란?

인터럽트(Interrupt)는 사전적 의미로 1.방해하다 2.중단시키다 라는 뜻을 가지고 있다. 하고 있는 일을 중단시키고 끼어드는 듯한 뉘앙스를 주는데 실제로 인터럽트는 cpu에게 일을 중단시키는 일을 한다. 인터럽트의 정확한 정의를 위키백과에서 찾아보았다.

“인터럽트란 CPU가 프로그램을 실행하고 있을 때, 입출력 하드웨어 등의 장치에 예외상황이 발생하여 처리가 필요할 경우에 CPU에게 알려 처리할 수 있도록 하는 것을 말한다”

풀어서 설명하자면 CPU가 어떤 일을 하다가 입출력 신호가 들어오면 입출력 관리자에게 입출력 명령을 보낸다. 그럼 입출력 관리자는 CPU대신 입출력 처리를 위한 데이터를 메모리에 가져다 놓거나 메모리에 있는 데이터를 저장장치로 옮긴다. 데이터 전송이 완료되면 입출력 관리자는 완료신호를 CPU에게 보낸다. 이 때 입출력 관리자가 CPU에게 보내는 완료 신호를 인터럽트라고 한다. 결론적으로 CPU가 인터럽트 신호를 보내는 것이 아니라 입출력 관리자가 CPU에게 인터럽트 신호를 보내는 것이다.

CPU는 사람으로 치자면, 매우 고급 인력이다. 그런 고급인력이 계속해서 들어오는 입출력 신호에 필요한 데이터를 찾아서 헤매는 것은 비효율적이다. 마치 이연복 요리사가 바쁜 점심시간에 주방에 재료가 떨어져 근처 슈퍼에서 직접 재료를 사오는 것과 비슷하다. 그런 오래걸리고 자잘한 일은 주방보조에게 시키는 것이 더 효율적일 것이다. 여기서 주방보조는 입출력 관리자와 같다. 입출력 관리자는 CPU가 입출력 신호를 받을 때 까지 대기하다가 CPU가 입출력 신호를 받으면 메모리에서 데이터를 가져와 CPU에게 전달하고 CPU는 하던 일을 멈추고 입출력 신호를 처리한다.

인터럽트 종류

사실 인터럽트는 키보드 마우스 입출력에만 국한되어 있지 않다. 인터럽트는 종류가 많고 매우 광범위하게 쓰인다. 예를 들어 프로그램에서 무한루프를 돌고있어 멈추지 않을 때 Control+ C 를 눌러 멈추는데 이것도 인터럽트의 한 종류이다. 인터럽트는 크게 내부 인터럽트 , 외부 인터럽트로 분류할 수 있다.

내부 인터럽트 내부 인터럽트는 프로그램 내에서 다음과 같이 잘못된 명령이나 데이터를 사용할 때 발생하며 트랩(Trap)이라고도 부른다. Zero division, Overflow, Underflow 등 런타임시 에러가 발생하면 인터럽트를 발동시킨다. 프로그램은 순서에 맞게 진행하다가 잘못된 명령어를 보고 인터럽트를 발생시키므로 동기적 인터럽트라고 불린다. 또는 에러 인터럽트, 소프트웨어 인터럽트라고도 불린다. 프로그래밍에서 예외처리를 한다고 하는데 그때 예외는 내부 인터럽트를 말한다.

외부 인터럽트 외부 인터럽트는 다음과 같이 입,출력장치, 타이밍 장치, 전원 등 외부적인 요인에 의해 발생한다.
전원 이상 인터럽트 : 정전이되거나 전원 이상이 있는경우 발생한다.

기계 착오 인터럽트 : CPU의 기능적인 오류 동작이 발생한 경우 발생한다.

외부 신호 인터럽트 : 타이머에 의해 규정된 시간을 알리는경우, 키보드로 인터럽트 키를 누른 경우, 외부 장치로부터 인터럽트 요청이 있는경우 발생한다.

입출력 인터럽트 : 입출력 Data의 오류나 이상 현상이 발생한 경우, 입출력장치가 데이터의 전송을 요구하거나 전송을 끝났음을 알릴경우 발생한다.

실행 중인 명령어와 무관하게 갑자기 발생하는 인터럽트라서 비동기적 인터럽트 또는 하드웨어 인터럽트 라고도 한다.

인터럽트 처리과정

인터럽트에는 해당 인터럽트가 발생하면 어떤 일을 할지도 미리 정의되어있다. 인터럽트 신호가 발생하면 미리 정의되어 있는 함수를 실행하고 작업을 다 마치면 원래 작업으로 돌아온다. 만약 인터럽트가 동시에 여러개 발생한다면 어떻게 될까? 운영체제는 동시에 발생할 인터럽트를 대비해 인터럽트 백터를 사용한다. 인터럽트 백터는 인터럽터들의 정의를 가리키는 주소를 저장하고 있는 자료구조이다. 인터럽트 백터에 들어온 여러신호를 보고 미리 정해둔 우선순위를 기반으로 인터럽트 작업들을 처리한다.

인터럽트 처리 과정을 자세히 정리하면 다음과 같다.

  1. 인터럽트가 발생하면 현재 실행 중인 프로세스는 일시 정지 상태가 되며, 재시작하기 위해 현재 프로세스 관련 정보를 임시로 저장한다.(PCB에 메모리주소, 현재 레지스터 값, 하드웨어 상태 등을 저장한다.)
  2. 인터럽트 컨트롤러가 실행되어 인터럽트의 처리 순서를 결정한다. 이때 여러 개의 인터럽트가 동시에 발생했다면 인터럽트 우선순위를 고려하여 중요한 인터럽트부터 처리하도록 순서를 결정한다.
  3. 먼저 처리할 인터럽트가 결정되면 인터럽트 벡터에 등록된 인터럽트 핸들러가 실행되어서 작업을 시작한다.
  4. 인터럽트 처리를 마치면 일시 정지된 프로세스가 다시 실행되거나 종료된다. (인터럽트가 입출력 완료 같은 경우이면 프로세스 재시작, 메모리 침범이나 오류같은 경우 프로세스 종료)

추가적으로 설명하자면 CPU는 명령어를 하나 수행할 때 마다 인터럽트 신호가 들어왔는지 동기적으로 체크한다. 그러므로 현재 수행중인 명령어에 영향을 주지 않는다.

직접메모리 접근

입출력 관리자는 CPU를 대신해서 메모리에 접근하여 데이터를 저장하거나 가져온다. 하지만 메모리는 CPU만 접근 가능한 작업 공간이라 입출력 관리자는 접근이 불가하다. 따라서 입출력 관리자는 메모리 접근 권한이 필요한데 이러한 권한을 직접 메모리 접근(DMA)이라고 한다.

메모리 맵핑 입출력

직접 메모리 접근은 인터럽트 방식의 시스템을 구성하는데 필수 요소이다. 그러나 직접 메모리 접근 방식을 사용하면 메모리가 복잡해진다. CPU가 사용하는 데이터와 입출력 장치가 사용하는 데이터가 섞여 있으면 데이터를 찾는 시간도 오래걸리고, 관리하기도 어렵다. 이를 막기위해 메모리를 나누어 사용하는 방법이 도입되었다. CPU가 사용하는 메모리 공간과 직접 메모리 접근 권한으로 들어온 데이터공간을 분리한 방식을 메모리 매핑 입출력(Memory Mapped I/O, MMIO)이라고한다.

사이클 훔치기

CPU와 직접메모리 접근이 동시에 메모리에 접근하면 누군가는 양보를 해야하는데 보통은 CPU가 메모리 사용 권한을 양보한다. CPU는 처리속도가 워낙 빠르기 때문에 상대적으로 느린 입출력 장치에게 순서를 양보한다. 이러한 상황을 사이클 훔치기(Cycle stealing)이라고 한다. CPU 입장에서는 직접메모리 접근이 사이클(순서)를 훔쳐 간 것이기 때문에 이렇게 일컫는 것이다.
