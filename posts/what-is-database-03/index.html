데이터베이스(3): 데이터베이스 구조와 언어

데이터베이스 구조

지금까지 DBMS의 등장배경과 종류에 대해 알아보았다. 지금부턴 데이터베이스의 구조에 대해 알아보겠다. 사실 데이터베이스의 내부는 복잡하다. 하지만 사용자 입장에서는 데이터베이스의 내부가 어떻게 생겼든 관심없다. 그냥 데이터만 잘 가져오면 된다. 이것을 추상화라고 표현한다. 반면 데이터베이스 관리자는 데이터를 어떻게 분류하고 정의할 지 고민해야 한다. 직접 데이터의 형식과 관계를 정의하고, 데이터베이스에 문제가 생기면 직접 해결 해야한다. 이렇게 어떤 입장이 되느냐에 따라 데이터베이스를 바라보는 관점도 달라진다.

스키마와 인스턴스

스키마 : 데이터베이스에 저장되는 데이터 구조와 제약조건을 정의한 것

예를 들어, 고객관리 데이터베이스에 어떤 항목을 넣을건지, 자료형은 무엇인지, null값을 허용할지 등등을 정의한것을 스키마라고 한다. 스키마는 나중에 변경하지 않는 이상, 변하지 않고 그 상태를 유지한다.

인스턴스 : 스키마에 따라 데이터베이스에 실제로 저장된 값

인스턴스는 고객관리 데이터베이스에서 한 명의 회원 데이터라고 생각하면 편하다. 그 회원의 전화번호나 나이는 바뀔 수 있다. 물론 삭제 될 수도있다. 즉 인스턴스는 계속변할 수 있다.

3단계 데이터베이스 구조

미국 표준화 기관인 ANSI/SPARC에서 제안한 데이터베이스 구조에 대해 알아보자.
데이터베이스를 쉽게 이해하고 이용할 수 있도록 하나의 데이터베이스를 관점에 따라 세 단계로 나누었다. 중요한 것은 관점에 따라 나눈것이다.

  • 외부단계(external level) : 개별 사용자 관점
  • 개념단계(conceptual level) : 조직 전체의 관점
  • 내부단계(internal level) : 물리적인 저장 장치의 관점

내부 단계에서 외부 단계로 갈수록 추상화 레벨이 높아진다. 추상화 레벨이 높아진다는 것은 데이터베이스가 실제 돌아가는 것에는 관심이 없다는 뜻과 비슷하다. 실제 사용자는 데이터베이스 내부 시스템을 알 수 없을 뿐만 아니라 알 필요도 없기 때문이다.

외부스키마

데이터베이스 하나에 외부 스키마가 여러 개 존재할 수 있다. 서브 스키마(sub schema)라고도 한다. 각 사용자가 생각하는 데이터베이스의 모습이라고 보면 된다. 고객 관리를 담당하는 직원은 고객 관련 데이터에만 관심을 갖고, 상품관리를 담당하는 직원은 상품과 관련된 데이터에만 관심을 갖는다. 결국 사용자에게 어떻게 보여지는가에 초점을 두기 때문에 ‘사용자 뷰’라고 부르기도 한다. 한 줄로 정의하자면 ‘사용자나 응용 프로그래머가 개인의 입장에서 필요한 데이터베이스의 논리적 구조를 정의’라고 할 수 있다.

개념스키마

데이터베이스를 조직 전체의 관점에서 이해하고 표현하는 단계이다. 데이터베이스 하나에 개념 스키마는 하나만 존재한다. 어떤 데이터가 저장되었는지, 데이터들 간에는 어떤 관계가 존재하고 어떤 제약조건이 있는지에 대한 정의 뿐만 아니라, 데이터에 대한 보안 정책이나 접근 권한에 대한 정의도 포함된다. 데이터베이스를 설계할 때 가장 첫번째로 하는 단계이다.

내부스키마

데이터베이스를 저장 장치의 관점에서 이해하고 표현하는 단계이다. 역시 데이터베이스 하나에 하나만 존재하고, 레코드 구조, 필드 크기, 레코드 접근 경로등 물리적 저장 구조를 정의한다. 시스템 설계자나 프로그래머가 바라보는 관점이다.

사상과 매핑

스키마는 하나의 데이터베이스를 논리적으로 구축하기 위해 만들어진 개념이라서 서로 잘 연결되어있다. 그렇기 때문에 스키마 사이의 대응 관계를 정의할 필요가 있다.
외부스키마와 개념스키마 사이의 대응 관계를 나타낸것을 외부/개념 사상이라고 한다. 또는 ‘응용 인터페이스’라고도 한다.

개념스키마와 내부스키마의 대응관계를 나타낸 것을 개념/내부 사상이라고 한다. 또는 ‘저장 인터페이스’라고 한다. 왜 스키마 사이의 대응 관계를 정의 했을까? 궁극적인 목적은 바로 이것이다.

” 데이터 독립성의 실현 ”

이것이 무슨 뜻인지 알아보자. 프로그램을 운영 또는 개발하다가 데이터베이스의 구조를 수정해야 할 때가 있을 것이다. 개념스키마에서 데이터 유효성검사를 수정해야 한다고 치자. 비밀번호를 6자리에서 8자리로 제약조건을 바꾼다 하더라도, 외부스키마는 바뀌지 않는다. 맵핑만 잘 해주면 된다. 즉 하위 스키마를 변경하더라도 상위 스키마가 영향을 받지 않는 특성을 데이터 독립성 이라고 한다.

논리적 데이터 독립성

  • 개념 스키마가 변경되어도 외부 스키마는 영향을 받지 않음
  • 개념 스키마가 변경되면 관련된 외부/개념 사상만 정확하게 수정해주면 됨

물리적 데이터 독립성

  • 내부 스키마가 변경되어도 개념 스키마는 영향을 받지 않음
  • 내부 스키마가 변경되면 관련된 개념/내부 사상만 정확하게 수정해주면 됨

스키마도 정의하고 매핑도 했으면 그 정보들을 기록할 필요가 있다. 이 정보들은 데이터 사전에 기록된다.

데이터 사전
스키마, 사상 정보, 다양한 제약조건 등을 저장
시스템 카탈로그 라고도 함
메타데이터
데이터베이스 관리 시스템이 스스로 생성하고 유지함
일반 사용자도 접근이 가능하지만 저장 내용을 검색만 할 수 있음

데이터베이스 언어

데이터베이스 언어는 말 그대로 사용자와 데이터베이스 관리 시스템 간의 통신 수단이다. 사용목적에 따라 데이터 정의어, 데이터 조작어, 데이터 제어어로 구분한다.

데이터 정의어

스키마를 정의하거나, 수정 또는 삭제하기 위해 사용한다.

데이터 조작어

데이터의 삽입, 삭제, 수정, 검색 등의 처리를 요구하기 위해 사용한다.

  • 절차적 데이터 조작어

    -사용자가 어떤 데이터를 원하고 그 데이터를 얻기 위해 어떻게 처리해야 하는지도 설명

  • 비절차적 데이터 조작어(선언적 언어)

    -사용자가 어떤 데이터를 원하는지만 설명

데이터 제어어

내부적으로 필요한 규칙이나 기법을 정의하기 위해 사용한다.

<사용목적>

  • 무결성 : 정확하고 유효한 데이터만 유지
  • 보안 : 허가받지 않은 사용자의 데이터 접근 차단, 허가된 사용자에 권한 부여
  • 회복 : 장애가 발생해도 데이터 일관성 유지
  • 동시성 제어 : 동시 공유 지원

데이터베이스 사용자

데이터베이스 사용자는 데이터베이스를 이용하기 위해 접근하는 모든 사람을 일컫는다. 이용 목적에 따라 데이터베이스 관리자, 최종 사용자, 응용 프로그래머로 구분한다.

데이터베이스 관리자

데이터베이스 시스템을 운영 및 관리하는 사람이다. 주로 데이터정의어와 제어어를 이용한다. 주요 업무로는 다음과 같다.

  • 데이터베이스 구성 요소 선정
  • 스키마 정의
  • 물리적 저장 구조와 접근 방법 결정
  • 무결성 유지를 위한 제약조건 정의
  • 보안 및 접근 권한 정책 결정
  • 백업 및 회복 기업 정의
  • 시스템 데이터베이스 관리
  • 시스템 성능 감시 및 성능 분석
  • 데이터베이스 재구성

최종 사용자

데이터베이스에 접근하여 데이터를 조작(삽입, 삭제, 수정, 검색)하는 사람이다.
주로 데이터 조작어를 사용한다. 직접 쿼리문을 사용하는 것이 아니라 GUI를 이용해 데이터를 조작한다.

응용 프로그래머

데이터 언어를 삽입하여 응용 프로그램을 작성하는 사람이다.
주로 데이터 조작어를 사용한다. 데이터를 이용해 프로그램을 만들고, 데이터를 잘 파싱해서 뿌려주는 역할을 한다.
